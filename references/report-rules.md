# 报告规则

报告是研发评审用初稿，不是最终立项决策。

每节至少包含：

1. 事实证据
2. 初步分析
3. 证据强度与可信度
4. 待确认事项 / 下一步补采建议
5. 支撑证据列表

不得把“未检索到”写成“不存在”。不得把采集失败写成“无证据”。没有证据支持的部分应标记为“证据缺口”。

## 研发筛选版布局

- 顶部页签：项目分析、研发阅读入口、指标事实、核心必读文献、全部证据卡、资料缺口。
- 项目分析保留可跳转的左侧 17 章目录；其他页签使用各自的局部筛选器。
- 每章依据表保留全部匹配结果，页面独立分页；数据层不得固定截断。
- 依据表展示原文标题链接、相关原文和可跳转的证据卡锚点。
- 研发阅读入口先给研发定位、验证重点、证据读数、可信度和下一关口。
- 指标事实独立成页，支持总查询和指标、数值、材料、证据卡、样本、平台/方法组合筛选。
- 核心必读文献和全部证据卡分别提供局部筛选器。

## 证据卡字段

证据卡展示优先级、研发阶段、样本、平台/方法、参照标准、应用场景、目标人群、性能参数、组合标志物、地域/人群、证据类型、关键摘录、中文阅读状态和待复核点。

## 翻译与渲染边界

- 翻译是显式前置步骤：先运行 `translation-status`，能力可用时再运行 `translate-materials`。
- 报告渲染只读取 `data/translations.jsonl` 缓存，不得创建翻译引擎、写翻译缓存或发起网络请求。
- HTML 优先显示中文标题和专业中文阅读，英文原文保留用于追溯。
- 来源、检索式、题名、作者、期刊和 Abstract 按信息块展示，不得挤成一个长段落。
- 翻译能力缺失时显示明确状态和安装提示，不得静默吞错。

## 业务可读缺口

采集异常、未命中、登录/权限受限应转换为“缺少什么、影响什么、由谁补充、如何验收”。不得向业务用户展示 HTTP、DOM、Playwright、adapter 或命令参数等内部实现细节。
